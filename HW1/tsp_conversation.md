
# TSP 與爬山演算法 — 問答對話紀錄

> 本文件紀錄學習旅行推銷員問題（TSP）與爬山演算法過程中的詳細問答。

---

## 對話紀錄
 
---

**我：** 旅行推銷員問題是什麼？

**AI：** 旅行推銷員問題（Travelling Salesman Problem，TSP）是組合優化領域最經典的問題之一。

問題定義很簡單：

> 給定 N 個城市以及每對城市之間的距離，找出一條路徑，使推銷員**從起點出發、拜訪每個城市恰好一次、最後回到起點**，且總距離最短。

聽起來直觀，但它是 **NP-Hard** 問題——城市數一多，可能的路徑組合會爆炸性成長：

| 城市數 N | 可能路徑數（(N-1)!/2） |
|---------|----------------------|
| 5       | 12                   |
| 10      | 181,440              |
| 20      | 約 6 × 10¹⁶         |
| 50      | 天文數字，暴力解不可能  |

---

**我：** 那爬山演算法是什麼？它怎麼套用在 TSP 上？

**AI：** 爬山演算法（Hill Climbing）是一種**局部搜尋**的最佳化方法，概念來自爬山：

1. 站在目前位置（current solution）
2. 看看附近（neighbor）哪個方向比較高
3. 走過去
4. 重複，直到四周都比自己低（local optimum）

套用在 TSP 上：

- **解（solution）**：一條拜訪所有城市的順序，例如 `[0, 3, 1, 4, 2]`
- **height**：評估這條路有多好，定義為 `−總距離`（距離越短，height 越高）
- **neighbor**：對目前路徑做一個小修改，產生一條相近的路徑

演算法流程：

```
初始解 → 計算 height → 產生 neighbor → 
  neighbor 比較好？
    Yes → 移動過去，繼續
    No  → 捨棄，再試一次
重複直到收斂或達到迭代上限
```

---

**我：** height 函數怎麼設計？

**AI：** 定義如下：

```python
def height(tour, cities):
    return -total_dist(tour, cities)
```

因為爬山演算法是「找極大值」，但我們想「最小化距離」，所以乘上 `-1` 做轉換：

```
距離越短  →  -距離越大  →  height 越高  →  爬山往這裡走 ✓
```

`total_dist` 計算方式是把路徑上所有相鄰城市的歐式距離加總：

$$\text{total\_dist} = \sum_{i=0}^{n-1} \sqrt{(x_{t_i}-x_{t_{i+1}})^2 + (y_{t_i}-y_{t_{i+1}})^2}$$

其中 $t_n = t_0$，最後一個城市接回起點。

---

**我：** neighbor 函數怎麼設計？可以選兩條邊 (a,b)(c,d)，然後鄰居變成 (a,d)(b,c) 嗎？

**AI：** 你描述的是 **2-opt** 操作，這是 TSP 最常用的鄰居定義，非常好的選擇！

不過這裡有個重要的數學細節要釐清——

在一個 cycle（環狀路徑）上，如果你刪掉兩條**不相鄰**的邊 `(a,b)` 和 `(c,d)`，整個 cycle 會斷成**兩條獨立的路徑**：

```
路徑 P1：b → ... → c
路徑 P2：d → ... → a
```

要把這兩條路徑重新接成一個合法的 Hamiltonian cycle，唯一的方式是**頭接頭、尾接尾**：

```
新邊：(a, c) 和 (b, d)
```

如果選擇 `(a,d)` 和 `(b,c)`：
- `(b,c)` 把 P1 的頭尾相接 → P1 自己變成一個子環
- `(a,d)` 把 P2 的頭尾相接 → P2 自己變成另一個子環
- **結果：兩個分開的環，不是合法解** ✗

所以 2-opt 的正確操作是產生 `(a,c)(b,d)`，實作上用**反轉中間段**來達成：

```
原路徑：... → a → [b → ... → c] → d → ...
                    ↑ 反轉這段 ↑
新路徑：... → a → [c → ... → b] → d → ...
```

```python
new_tour = tour[:i+1] + tour[i+1:j+1][::-1] + tour[j+1:]
```

---

**我：** 初始解要怎麼設定？

**AI：** 最簡單的初始解是「照城市編號依序走」：

```python
current = list(range(n))  # [0, 1, 2, 3, ..., n-1]
```

表示路徑：`0 → 1 → 2 → 3 → ... → n-1 → 0`

這個初始解不需要任何計算，但通常很差（因為城市編號和地理位置沒有關係），距離可能很長。這反而是好事——能讓我們清楚看到爬山演算法的**改善幅度**有多大。

---

**我：** 爬山演算法的主迴圈怎麼寫？

**AI：** 核心很簡單：

```python
def hill_climbing(cities, max_iter=100_000):
    n = len(cities)
    current = list(range(n))
    cur_h   = height(current, cities)

    improved = 0
    for _ in range(max_iter):
        nbr   = neighbor(current)
        nbr_h = height(nbr, cities)
        if nbr_h > cur_h:          # 嚴格改善才走過去
            current, cur_h = nbr, nbr_h
            improved += 1

    return current, -cur_h, improved
```

每次迭代：
1. 呼叫 `neighbor()` 隨機產生一個 2-opt 鄰居
2. 計算鄰居的 height
3. 比較：只有比現在更好才接受
4. 跑滿 `max_iter` 次停止

---

**我：** 為什麼 neighbor 選 i 的時候要限制到 `n-3` 而不是 `n-2`？

**AI：** 因為 `j` 的範圍是 `random.randint(i+2, n-1)`，需要 `i+2 <= n-1`，即 `i <= n-3`。

如果 `i = n-2`，則 `i+2 = n`，但 `j` 最大只能是 `n-1`，會造成 `randint(n, n-1)` 這種空範圍錯誤：

```
ValueError: empty range in randint(20, 19)
```

所以 `i` 的上界是 `n-3`，確保 `j` 至少有一個合法值可選。

---

**我：** 還有為什麼要特別排除 `i=0, j=n-1` 這組？

**AI：** 因為這組選到的兩條邊是：

```
edge1 = (tour[0], tour[1])
edge2 = (tour[n-1], tour[0])   ← 這條邊就是繞回起點的邊
```

`edge2` 的終點 `tour[(n-1+1) % n] = tour[0]` 就是 `edge1` 的起點，表示這兩條邊**在 cycle 上其實是相鄰的**。反轉 `tour[1..n-1]` 只是把整條路倒著走，由於 TSP 是無向圖，倒著走距離完全相同，等於白做。所以跳過這組。

---

**我：** 爬山演算法有什麼缺點？怎麼改進？

**AI：** 最大缺點是**容易卡在 local optimum**。

爬山只往「當下看到的更高處」移動，一旦四周都比自己低，就永遠停在那裡——即使全域最優解在更遠的地方。

常見改進方法：

| 方法 | 概念 | 效果 |
|------|------|------|
| **隨機重啟（Random Restart）** | 多跑幾次不同初始解，取最佳結果 | 簡單有效，但沒有本質突破 |
| **模擬退火（Simulated Annealing）** | 以一定機率接受較差的解，讓搜尋能「跳坑」 | 能跳出 local optimum，品質大幅提升 |
| **基因演算法（GA）** | 模擬生物進化，多個解互相交叉、突變 | 適合大規模問題 |
| **蟻群演算法（ACO）** | 模擬螞蟻留費洛蒙尋找最短路 | TSP 專屬，效果很好 |

---

**我：** 跑出來的結果如何？

**AI：** 以 20 個城市、seed=42、max_iter=100,000 為例：

```
城市數:   20
初始距離: 1090.33
最短距離: 452.50
改善幅度: 58.5%
接受改善: 35 次
最佳路徑: 0 → 4 → 13 → 6 → 5 → 8 → 16 → 15 → 10 → 18 → 14 → 19 → 12 → 2 → 7 → 17 → 1 → 11 → 3 → 9 → 0
```

初始解 1090 → 爬山後 452，改善約 **58.5%**。

值得注意的是，100,000 次迭代中只有 **35 次**真正接受改善——大部分的時間都在嘗試但失敗。這說明越到後期，找到更好鄰居的機率越低，搜尋空間已接近（局部）最優。

---

**我：** 所以整體來說，爬山演算法解 TSP 的流程是？

**AI：** 總結如下：

```
1. 初始解  →  current = [0, 1, 2, ..., n-1]

2. 評估    →  cur_h = height(current) = -total_dist(current)

3. 迭代（重複 max_iter 次）：
   a. 隨機選兩個索引 i < j（且 j >= i+2）
   b. 反轉 tour[i+1..j]，得到 neighbor
   c. 計算 nbr_h = height(neighbor)
   d. 若 nbr_h > cur_h：接受 neighbor 為新的 current

4. 回傳最終 current 與對應的最短距離
```

簡單、快速，適合作為更複雜演算法的基礎或比較基準。
